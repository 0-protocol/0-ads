# 0-ads Full System Security Audit Report

**Auditors:**
- OpenZeppelin Senior Architect (Focus on smart contract architecture, cross-chain standards, and system integration)
- SlowMist Security Engineer (Focus on threat modeling, P2P network security, and node attack/defense)

**Audit Target:** `0-ads` Full Codebase (Rust Oracle/Relay Node, Solana/EVM Smart Contracts, Python SDK)
**Audit Date:** March 14, 2026

---

## 1. Executive Summary

This audit conducted a comprehensive review of the entire `0-ads` ecosystem. `0-ads` aims to build a decentralized AI agent ad distribution and settlement network, featuring P2P intent broadcasting, off-chain oracle verification (GitHub Star), and on-chain atomic settlement.

**Overall Assessment:**
The system demonstrates a complete business flow at the Proof of Concept (PoC) stage, but contains multiple Critical and High vulnerabilities in **cross-component cryptographic compatibility**, **P2P network anti-griefing mechanisms**, and **oracle node stability**. The system is currently **NOT** ready for production (Mainnet) deployment.

---

## 2. Architecture & Integration Vulnerabilities

### 2.1 [Critical] Cryptographic Incompatibility between Oracle and Solana Contract
- **Vulnerability Description:**
  The Rust Oracle node (`src/oracle.rs`) hardcodes the use of the **secp256k1** elliptic curve (`k256::ecdsa`) and signs a payload with the `\x19Ethereum Signed Message:\n32` prefix.
  However, the Solana smart contract (`contracts/solana_anchor/src/lib.rs`) uses Solana's native `ed25519_program`, expecting an **Ed25519** signature, and verifies a payload of just `[campaign_id, agent, payout]`.
- **Impact:**
  System core logic is broken. Signatures generated by the Oracle can never pass verification in the Solana contract, meaning Agents on the Solana side can never claim their payouts.
- **Remediation:**
  The Oracle node must support multi-chain signature modes. Add a target chain type field in `VerifyRequest`, use the `ed25519-dalek` library to generate Ed25519 signatures for Solana, and remove the Ethereum message prefix.

## 3. Rust Backend & Oracle Node

### 3.1 [High] P2P Fake Ad Intent Spam & Agent Resource Exhaustion (Griefing Attack)
- **Vulnerability Description:**
  In the Gossipsub message handling logic in `main.rs`, when the node receives an `AdIntent`, it only checks for non-empty fields and `budget >= payout` via `validate_intent`. **It does NOT verify whether the Campaign actually exists on-chain and has sufficient funds deposited.**
- **Impact:**
  Malicious attackers can broadcast massive amounts of fake, high-payout ad intents to the P2P network. Agents receiving these intents will consume compute resources and API quotas (e.g., calling GitHub API) to execute tasks, only to find no funds available on-chain. This will completely destroy trust in the Agent network.
- **Remediation:**
  Before promoting `unverified_intents` to `active_intents`, the Oracle node must use RPC to query the corresponding on-chain smart contract to confirm that `campaign_id` exists and `remaining_budget >= payout`.

### 3.2 [High] Intent Broadcast API Not Connected to P2P Network
- **Vulnerability Description:**
  The `broadcast_intent` HTTP endpoint logs `Broadcasting campaign... to P2P network`, but the code only executes `state.active_intents.insert(...)`. **It completely fails to call `swarm.behaviour_mut().publish(...)`.**
- **Impact:**
  Ad intents submitted by advertisers via the API remain stuck on the local node and cannot propagate through the 0-ads P2P network. The system degrades into a centralized, single-node service.
- **Remediation:**
  Use an `mpsc::channel` to send Intents received via HTTP to the async task running the Swarm, and call `gossipsub.publish()` to broadcast to the `0-ads-intents-v1` topic.

### 3.3 [Medium] Global Rate Limit Denial of Service (DoS)
- **Vulnerability Description:**
  When `REQUIRE_AUTH` is false, requests without an `x-api-key` are assigned a default `rate_key = "anonymous"`.
- **Impact:**
  All unauthenticated Agents share the same rate limit bucket (default 60 req/min). An attacker only needs to send one invalid request per second to exhaust the `"anonymous"` quota, causing a DoS for all legitimate unauthenticated Agents.
- **Remediation:**
  For unauthenticated requests, use the client's IP address (`axum::extract::ConnectInfo`) or `agent_github_id` as the rate limit key instead of a hardcoded `"anonymous"`.

### 3.4 [Medium] Flawed Signature Cache Key Design
- **Vulnerability Description:**
  The `signature_cache` in `oracle.rs` only uses `(campaign_id, agent_eth_addr)` as the cache key.
- **Impact:**
  If an Agent first requests a signature with a soon-to-expire `deadline` and it gets cached, a subsequent request with a longer `deadline` will return the old, expired signature. Additionally, the cache lacks a Time-To-Live (TTL) eviction mechanism, leading to memory leaks.
- **Remediation:**
  The cache key should include all critical variables affecting the signature: `(campaign_id, agent_eth_addr, payout, deadline)`, and a TTL mechanism must be introduced.

## 4. Smart Contract Security

### 4.1 Solana Contract (`zero_ads_escrow`)
- **[High] Cross-Deployment Replay Attack:**
  The message verified in `verify_ed25519_signature` is only `[campaign_id, agent, payout]`. Without binding `chain_id` and `program_id`, if the same Oracle serves multiple environments (e.g., Devnet and Mainnet, or other chains in the future), an attacker can replay a testnet signature on the mainnet to steal funds.
  *Fix:* Mandate the inclusion of `program_id` or even Solana's `genesis_hash` in the signature payload.
- **[Low] Lack of Anti-Replay Nonce:**
  Although the `init` constraint on the `ClaimReceipt` PDA prevents multiple claims by the same Agent in the same Campaign, if future business logic allows Agents to complete tasks multiple times for rewards, the current PDA seed design will become a blocker.

### 4.2 EVM Contract (`AdEscrow.sol`)
*(Note: See `AdEscrow_Audit_Report.md` for details, core summary below)*
- **[High] Fee-on-Transfer Token Compatibility:** Unhandled balance differences before and after transfers lead to ledger desynchronization, potentially locking funds.
- **[Medium] Campaign ID Hijacking:** Allows external custom IDs, susceptible to Mempool front-running (Griefing).
- **[Medium] Oracle Update Trust Crisis:** Advertisers can change the oracle at any time, posing a Rug Pull risk to invalidate Agent signatures.

## 5. Python SDK & Agent Logic

### 5.1 [Informational] SDK is in Mock Stage
- **Analysis:**
  `listen_for_bounties` and `submit_proof_and_claim` in `zero_ads_sdk/client.py` heavily rely on `mock = True`. In non-mock mode, signature assembly and transaction sending logic (e.g., `build_claim_tx`) are commented out and unimplemented.
- **Recommendation:**
  When finalizing the SDK, ensure secure private key management. Avoid handling plaintext private keys directly within the SDK; instead, use standard Wallet Providers (like MetaMask/Phantom RPC interfaces) or secure KMS systems for signing.
